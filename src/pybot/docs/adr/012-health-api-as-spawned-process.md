# ADR 012: Управление процессом Health API в рамках Splitted Monolith

## Дата: 19-04-2026

## Статус: Deprecated

> Superseded by [ADR 0014: Health API как отдельный process type через Docker Compose и Uvicorn](014-health-api-as-dedicated-process-type.md).

## Контекст

Проект спроектирован по методологии 12 Factor App и разделяет логические пулы задач (бот, воркер TaskIQ, планировщик TaskIQ) на разные контейнеры. Однако, мы реализовали Health API (мини-сервер, обрабатывающий Liveness и Readiness запросы) как дочерний процесс внутри контейнера `bot`. Запуск происходит через `multiprocessing.spawn()` в точке входа `run.py`.

Согласно строгим рекомендациям 12 Factor App (Factor VI: Processes), каждый контейнер в идеале должен управлять ровно одним процессом. Выделение Health API в отдельный сервис (например, самостоятельный aiohttp/FastAPI контейнер) выглядит правильным шагом в сторону "инфраструктурной чистоты". Однако такое физическое разделение сильно усложнило бы деплой, так как Health API логически неразрывно связан с состоянием основного процесса бота и требует тех же зависимостей для проверки базы и кэша.

## Решение

Мы принимаем решение **оставить запуск Health API в виде дочернего `spawn`-процесса внутри контейнера бота**.

Мы рассматриваем архитектуру проекта как "Splitted Monolith" — единый монолит, функционал которого разделен только в местах, где требуется изоляция ресурсов (очереди и планировщик). Вынос Health API в отдельный независимый контейнер сместил бы архитектуру проекта в сторону классических микросервисов. На текущем этапе жизненного цикла проекта это очевидно излишне и принесет системный overhead.

### Детали реализации

1. Скрипт `run.py` остаётся входной точкой контейнера приложения бота. Основной код поднимает контекст бота, а перед этим `spawn`-ит независимый процесс сервера Health API.
2. В `run.py` предусмотрена обработка завершения: блок `finally` гарантирует отправку `process.terminate()` и ожидание завершения (`process.join()`) дочернего процесса Health API при смерти основного процесса бота.
3. Процесс Health API запускается на отдельном ядре (в теории изолированно), но разделяет сетевое пространство того же Docker-контейнера и светит портом для оркестратора (Docker/Kubernetes).

## Альтернативы

* **Вариант 1: Вынос Health API в отдельный контейнер в Docker Compose:**
  * *Плюсы:* Идеальное следование strict 12-Factor (1 процесс = 1 контейнер).
  * *Минусы:* Создаёт классический микросервисный оверхед. Увеличивает использование памяти, так как требует загрузки второго Python-интерпретатора, всех Pydantic-моделей и всего графа DI-зависимостей только ради отдачи HTTP-статуса `200 OK`.

* **Вариант 2: Запуск Health API корутиной внутри Event Loop самого бота (без `multiprocessing`):**
  * *Плюсы:* Идеально встраивается в монолит, один процесс, нет RAM оверхеда.
  * *Минусы:* Любые CPU-bound операции или временные блокировки асинхронного цикла бота (отправка больших файлов, спайк трафика) приведут к тому, что Liveness Probe от оркестратора завершится по таймауту. В итоге Docker ошибочно расценит занятый (но живой) бот как мертвый и перезапустит здоровый контейнер. Спавн нового процесса решает эту проблему изоляцией CPU-ресурсов.

## Последствия

### Положительные

* [+] Единый контейнер бота упрощает деплой, логирование и отладку (краши Health API видны рядом с логами бота).
* [+] Мы сохраняем высокую надежность оркестрации: изоляция через OS-уровень гарантирует, что Health API всегда отзывчив на Heartbeat-запросы, даже если цикл бота заблокирован.
* [+] Сохраняется парадигма "Масштабируемый Монолит", команда не тратит время на поддержку микросервисной инфраструктуры.

### Отрицательные

* [-] Технически мы нарушаем идеализированное толкование правила "Factor VI: Processes", пряча два демона в одном контейнере.
* [-] Использование `multiprocessing` обязывает аккуратно обращаться с обработкой сигналов `SIGTERM`/`SIGINT`, чтобы не порождать зомби-процессы внутри контейнера при оркестрации. (Частично нивелировано в `run.py`).
