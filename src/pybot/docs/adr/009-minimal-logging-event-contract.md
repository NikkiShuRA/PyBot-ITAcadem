# ADR 0009: Минимальный контракт событийного логирования

## Дата: 19-03-2026

## Статус: Accepted

## Контекст

С переходом проекта от локального интерактивного запуска к контейнеризации и deploy на VDS логи стали основным источником оперативной диагностики. При этом у проекта пока нет внешней observability-платформы, JSON-агрегатора или полноценной системы метрик.

Без зафиксированного минимума лог-событий формат логов начинает расползаться: разные модули пишут разные сообщения, поля появляются и исчезают, а grep и ручной анализ становятся менее надёжными.

## Решение

Принято решение использовать минимальный контракт событийного логирования поверх уже внедрённого key-value подхода.

### Детали реализации

1. Базовый формат логов: человекочитаемый `key=value`, на русском языке, без emoji и без JSON-sink в основном runtime.
2. Для core runtime фиксируется минимальное ядро событий:
   - `инициализация_бота`
   - `запуск_polling`
   - `graceful_shutdown`
   - `получен_update`
   - `обработан_update`
   - `ошибка_handler`
   - `медленный_handler`
   - `инициализация_taskiq_worker`
   - `завершение_taskiq_worker`
   - `отправка_уведомления`
   - `неожиданная_ошибка_runtime`
3. Если событие относится к обработке update или операции с измеримой длительностью, по возможности сохраняются поля `event_id`, `status`, `handler`, `elapsed_ms`, `user_id`.
4. Feature-level и временные debug-логи разрешены, но они не должны ломать или переопределять смысл ядра событий.

## Альтернативы

- **Не фиксировать контракт вообще:**
  - *Плюсы:* Ноль дополнительной документации и ограничений.
  - *Минусы:* Логи быстро теряют однородность, сложнее сопровождать runtime и писать алерты поверх логов.

- **Сразу вводить полноценную JSON-schema или внешнюю observability-платформу:**
  - *Плюсы:* Более сильная база для агрегации, метрик и алертинга в будущем.
  - *Минусы:* Избыточно для текущего этапа проекта, повышает сложность и не окупается без внешнего потребителя.

## Последствия

### Положительные

- [+] Логи остаются предсказуемыми между релизами и проще анализируются вручную.
- [+] Повышается устойчивость runtime-диагностики без тяжёлой инфраструктуры.
- [+] Поверх такого контракта проще добавлять лёгкие alerts и локальные ops-отчёты.

### Отрицательные

- [-] Разработчикам нужно сверяться с ядром событий при добавлении новых runtime-логов.
- [-] Контракт придётся периодически расширять по мере появления новых подсистем.
