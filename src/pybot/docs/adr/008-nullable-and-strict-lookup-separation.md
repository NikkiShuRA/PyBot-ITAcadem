# ADR 0008: Чёткое семантическое разделение между nullable и strict lookup

## Дата: 12-03-2026

## Статус: Accepted

## Контекст

Некоторые из методов Репозитория и application services, несмотря на None в type hints их возвращаемых значений, на самом деле возвращали ошибку при отсутствии искомого значения. Таким образом функции, выдававшие себя за nullable lookup, на самом деле являлись strict lookup, что плодило лживые и лишние проверки на None, которые методы всё равно вернуть не могли.

## Решение

Принято решение чётко семантически разделить nullable и strict lookup, а также исправить методы с некорректными type hints, убрав лишние проверки на None в местах, где они используются.

Также была принята следующая конвенция для избежания подобной ситуации в будущем:

`find_*` — возвращают None, если отсутствие сущности является ожидаемым сценарием.

`get_*` — поднимают domain error, если сущность обязательна для продолжения use-case.

Методы не должны одновременно:

- аннотироваться как Optional[...]
- и при этом бросать ...NotFoundError как основной путь отсутствия данных.

После strict lookup в коде не должно быть `if entity is None`.

При работе с коллекциями:

- `find_*` возвращает пустую коллекцию;
- `get_*` на коллекциях поднимает ошибку.

Таким образом:

- single optional entity -> find_* -> T | None
- single required entity -> get_* -> T или domain error
- many optional entities -> find_* -> Sequence[T]
- many required entities -> get_* -> Sequence[T] или domain error
- existence check -> exists_*/ has_* -> bool
- command/update methods -> не возвращают None как сигнал ошибки

Правило в первую очередь относится к repository и application service contracts, для прочих элементов системы это правило должно рассматриваться в частном порядке или в другом ADR.

### Детали реализации

1. Исправлена сигнатура методов `get_user` и `remove_user_role` через удаление `None` из возвращаемых значений
2. Были удалены все бесполезные проверки на `None` из мест, где используются вышеуказанные методы и заменены на try/except с re-raise ошибок с сохранением контекста.
3. Переименование методов `get_user_by_telegram_id` и `get_user_by_phone` согласно конвенции, чтобы уточнить их nullable lookup-природу

```python
# Пример реализации (если нужно)
    async def find_user_by_phone(
        self,
        db: AsyncSession,
        phone: str,
    ) -> User | None:
        ...

    async def find_user_by_telegram_id(self, tg_id: int) -> UserReadDTO | None:
        ...

    async def get_user(
        self,
        user_id: int,
    ) -> UserReadDTO:
        ...

    async def remove_user_role(self, tg_id: int, role_name: str) -> UserReadDTO:
        ...

```

## Альтернативы

- **Оставить всё как есть:**
  - *Плюсы:* Ничего менять не нужно
  - *Минусы:* Лишние проверки на None, вводящая в заблуждение семантика и вероятность дальнейшего расползания подобного поведения

- **Изменение поведения, без разделения семантики:**
  - *Плюсы:* Исправление вводящей в заблуждение семантики type hints и удаление лишних проверок на None
  - *Минусы:* Возможность повторения подобной ситуации в дальнейшем, если у нас не будет чёткого определения что стоит использовать при нейминге nullable и strict lookup логики, что может привести к повторению путаницы между ними.

## Последствия

### Положительные

- [+] Улучшение качества кода за счёт чёткого разделения ответственности и удаления бесполезных проверок
- [+] Чёткое семантическое разделение между двумя схожими подходами к поиску элементов или коллекций

### Отрицательные

- [-] Необходимость чётко в коде разделять проверки между nullable и strict lookup
- [-] Необходимость сверяться с конвенцией при создании новых методов application services и Репозиториев
