# ADR 0013: Границы конфигурации, fail-fast и явная инъекция настроек

## Дата: 20-04-2026

## Статус: Accepted

## Контекст

Во время крупного изменения вокруг конфигурации проект столкнулся с повторяющимися падениями CI на этапе import-time:

- валидация `BotSettings` срабатывала при простом импорте модулей;
- module-level чтение `settings` в декораторах, глобалах и runtime-модулях вызывало побочные эффекты;
- TaskIQ-часть местами зависела от скрытой регистрации задач и динамических импортов;
- тесты становились хрупкими из-за патчей глобального `module.settings`.

При этом для проекта критично сохранить:

- fail-fast/fail-loud валидацию обязательных env через Pydantic;
- текущую архитектуру c Dishka DI и composition roots;
- рабочий graceful shutdown и lifecycle hooks runtime-процессов.

Нужен был единый, формализованный контракт: где и когда конфигурация может читаться, где допустим fail-fast, и как избежать скрытых import-time side effects без полного Big Bang-рефактора всего проекта за один шаг.

## Решение

Принято решение использовать **явные runtime-границы конфигурации с `BotSettings + get_settings()`**, сохраняя fail-fast, но убирая неявные import-time побочные эффекты из feature-модулей.

### Детали реализации

1. Базовый контракт конфигурации:
   - `BotSettings` остается единственным источником типизированной конфигурации.
   - `get_settings()` остается точкой доступа к конфигурации и кэшируется (`lru_cache`).
   - fail-fast на отсутствующие обязательные переменные окружения сохраняется как целевое поведение.

2. Границы fail-fast:
   - fail-fast обязателен в composition root/bootstrap/runtime entrypoint.
   - обычный импорт feature-модуля не должен иметь скрытых побочных действий (создание broker/engine, чтение env через глобалы в декораторах и т.д.).

3. Правило anti-side-effects:
   - запрещается module-level доступ к `settings`, если он влияет на регистрацию роутеров/тасок или runtime-инициализацию.
   - настройки передаются явно в конструкторы/фабрики/функции регистрации.

4. DI и wiring:
   - для сервисов/адаптеров используется constructor injection `BotSettings`.
   - wiring выполняется в контейнерах и composition roots, а не внутри бизнес-модулей.
   - утилиты, где возможно, переводятся на pure API с явными параметрами (без чтения глобального `settings`).

5. TaskIQ-контракт:
   - регистрация задач должна быть явной и централизованной (`register_tasks` / `register_all_tasks`).
   - runtime не должен полагаться на скрытые module-level регистрации и цепочки `importlib` для побочных эффектов.
   - graceful shutdown и worker hooks не меняются в рамках этого решения.

6. Стратегия внедрения:
   - изменения идут поэтапно (small slices), после каждого этапа обязателен зеленый `just quality-gate`.
   - приоритет: предсказуемость runtime и тестируемость, без тотального одномоментного переворота архитектуры.

```python
# Принципиальный пример: настройки передаются явно в composition root.
runtime_settings = get_settings()
setup_dispatcher(runtime_settings)
setup_middlewares(dp, runtime_settings)
```

## Альтернативы

- **Полный откат изменений по конфигурации к прежней схеме с глобальным `settings`:**
  - *Плюсы:* меньше immediate-изменений, быстрый возврат к знакомому виду.
  - *Минусы:* возвращение скрытых зависимостей, import-time побочек и хрупких тестовых патчей.

- **Big Bang-рефактор всего проекта за один PR:**
  - *Плюсы:* единообразие сразу во всех модулях.
  - *Минусы:* высокий риск регрессий, сложный review, тяжелая диагностика и откат.

- **Повсеместные proxy-обертки вокруг настроек:**
  - *Плюсы:* временная совместимость API.
  - *Минусы:* рост абстракций без достаточной пользы, неочевидные точки materialization настроек, усложнение отладки.

## Последствия

### Положительные

- [+] Контракт по конфигурации стал явным: где допустим fail-fast и где недопустимы import-time side effects.
- [+] Меньше скрытых глобальных зависимостей, проще тестировать без `patch("...settings...")`.
- [+] CI и bootstrap-поведение стали более предсказуемыми.
- [+] DI используется по назначению: зависимости объявляются в конструкторах, wiring живет в composition root.

### Отрицательные

- [-] Увеличился объем явного wiring и число параметров в конструкторах/фабриках.
- [-] Переход требует дисциплины и последовательного рефактора по срезам, а не разовых правок.
- [-] Часть legacy-модулей может временно оставаться смешанной (старый и новый стиль), пока этап миграции не завершен.
